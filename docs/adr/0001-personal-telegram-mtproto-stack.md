# ADR 0001: Стек MTProto для личного аккаунта (TDLib vs Telethon/Pyrogram)

## Статус

Принято (черновик архитектуры).

## Контекст

Нужен сервис синхронизации **личного** Telegram-аккаунта (не Bot API): диалоги, сообщения, отправка от имени пользователя. Агент (OpenClaw) не должен иметь прямого доступа к MTProto — только к **внутреннему HTTP API** с политиками и аудитом.

## Решение: рекомендуемый стек для первой реализации

### Вариант A — **TDLib** (рекомендован для production-коннектора)

| Плюсы | Минусы |
|--------|--------|
| Официальный клиентский слой Telegram, актуальная совместимость | Сборка нативного модуля, CI/CD сложнее |
| Стриминг обновлений, стабильная модель состояния | FFI/обвязка (Node `tdl`, Python `python-telegram`, отдельный sidecar) |
| Хорошо для долгоживущего процесса «один аккаунт — один воркер» | Документация размазана по JSON API |

**Когда выбирать:** нужен **надёжный** долгоживущий sync, готовность поднять **отдельный процесс** (sidecar) с TDLib и говорить с ним по JSON-RPC/локальному сокету.

### Вариант B — **Telethon или Pyrogram** (рекомендован для быстрого MVP и прототипа)

| Плюсы | Минусы |
|--------|--------|
| Быстрый старт на Python, async, понятная модель | Зависимость от поддержки библиотеки при изменениях MTProto |
| Удобно для скриптов, бэкофиса, ручного теста | Менее «официально», чем TDLib, для enterprise иногда смотрят скептически |
| Легко вынести в отдельный микросервис-коннектор | Нужно аккуратно про лимиты и flood |

**Когда выбирать:** **MVP за короткий срок**, команда сильнее в Python; позже можно **заменить** реализацию коннектора на TDLib, сохранив те же границы API и БД.

## Итоговая рекомендация для VPN_service

1. **Коннектор** — **отдельный сервис** (`telegram-user/connector` → в будущем выделенный репозиторий или контейнер):  
   - **MVP:** Python + **Pyrogram** или **Telethon** (один выбрать и зафиксировать в ISSUES).  
   - **v1 production:** рассмотреть миграцию на **TDLib sidecar** с тем же контрактом очереди/БД.

2. **Граница:** коннектор пишет в **общую БД** (или в очередь → воркер пишет в БД), **не** отдаёт MTProto наружу.

3. **API gateway** — Node/Express или Fastify в монорепо, отдельный порт; авторизация **service JWT** + **policy engine** (заглушки в коде).

## Последствия

- Зависимость от выбранной библиотеки; смена на TDLib — **замена коннектора**, без ломки agent API при стабильном контракте.
- Python-коннектор потребует в `docker-compose` отдельного образа или venv в CI.
